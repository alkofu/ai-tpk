# Slash Commands

This document lists all Claude Code slash commands installed by this repository and what each one does.

## Commands

Claude Code slash commands provide quick workflow shortcuts. Commands are installed alongside skills and agents into `~/.claude/commands/`.

| Command | Purpose |
|---------|---------|
| `/bug` | Report a bug or investigate unexpected behavior — routes directly to Tracebloom (Investigative Gate), bypassing heuristic task classification. |
| `/bug-issue <issue-ref>` | Like `/bug`, but fetches the task description from a GitHub issue first. Accepts a bare issue number (e.g. `42`) or a full github.com issue URL (GitHub Enterprise URLs are not supported). Requires the `gh` CLI to be installed and authenticated. The DM runs `gh issue view` and constructs the task description from the returned title, body, labels, and comments before routing to Tracebloom. The task description includes a `Fixes #<number>` line immediately after the header, which propagates into commit messages and the PR body so GitHub auto-closes the linked issue when the PR is merged. When comments exist, a `## Comments` section is appended after the issue body so Tracebloom sees the full discussion thread. |
| `/feature` | Request a new feature or enhancement — routes directly to the constructive planning pipeline, bypassing the Investigative Gate. Pass `--docs` anywhere in your message (e.g., `/feature --docs fix typo in CONTRIBUTING.md`) to skip Pathfinder's requirements interview and scope-confirmation steps for self-evidently documentation-only tasks. All quality gates (Section 5 plan-confirmation, Phase 2 plan review, Phase 4 implementation review) still run. See the `--docs` row in DM's Workflow Flags table for the full definition. |
| `/feature-issue <issue-ref>` | Like `/feature`, but fetches the task description from a GitHub issue first. Accepts a bare issue number or full github.com issue URL (GitHub Enterprise URLs are not supported). Requires the `gh` CLI to be installed and authenticated. The DM runs `gh issue view` and constructs the task description from the returned title, body, labels, and comments before entering the Intake Gate (Askmaw or Pathfinder). The task description includes a `Closes #<number>` line immediately after the header, which propagates into commit messages and the PR body so GitHub auto-closes the linked issue when the PR is merged. When comments exist, a `## Comments` section is appended after the issue body so downstream agents see the full discussion thread. Pass `--docs` anywhere in your message (e.g., `/feature-issue --docs 42`) to skip Pathfinder's requirements interview and scope-confirmation steps for self-evidently documentation-only tasks. All quality gates (Section 5 plan-confirmation, Phase 2 plan review, Phase 4 implementation review) still run. See the `--docs` row in DM's Workflow Flags table for the full definition. |
| `/draft-issue <description>` | Converts a short feature description into a structured GitHub issue using the `action-item.md` template, then delegates to Bitsmith to create it via `gh issue create`. Runs in the advisory pipeline — no worktree, no plan file, no branch. DM asks one to three clarifying questions when the description leaves load-bearing gaps, then synthesises a title, body, and label set (always `enhancement`, plus any applicable `review:*` specialist-routing labels). DM presents the proposed issue inline and waits for explicit confirmation before filing. On success, echoes the new issue URL. Requires the `gh` CLI to be installed and authenticated. The user can later run `/feature-issue <url>` to start the constructive planning pipeline against the created issue. |
| `/ask` | Ask a question about the codebase, architecture, or approach — lightweight Q&A with no planning or implementation. Routes to the Advisory Workflow (Phases A-B-C). When the session surfaces an action item, capture it with the "Action item" issue template and hand off to `/feature-issue`. When the session produces multiple action items, open a "Session parent" issue first to group them, then file each action item as a sub-issue linked from the parent. |
| `/ops` | Runs an advisory query and saves the synthesis output as a Markdown report to `reports/` in the current repo. Thin alias for `/ask --save-report`. |
| `/do <action>` | Executes an operational write action via the advisory pipeline. No worktree, no plan file, no Ruinor review. Two execution paths share the same entry point: **(1) Single-command path** — when the user's prose resolves to one allowlist-conforming `gh` CLI command (e.g., `gh issue label 42 bug`), DM presents the exact command for explicit confirmation and delegates a single-shot execution to Bitsmith; outcome is reported inline. **(2) Multi-step fallthrough path (v1: GitHub issue/PR iteration only)** — when the prose requires sequencing or iteration over a GitHub issue or pull-request result set that cannot be expressed as a single allowlist-conforming command, DM falls through to a multi-step delegation flow: shows the user's prose verbatim with a safeguards summary, requires typed `CONFIRM` (case-insensitive) for the entire task, then delegates to Bitsmith with a repo-qualified locked item set (items are recorded as `(owner, repo, number)` tuples — write attempts targeting a different repo halt the task) and a locked `gh` write subcommand. Bitsmith first runs a read-only enumeration and reports the item list; DM enforces a 50-item cap before any writes proceed, and rejects outright if the item count exceeds 200 (no CONFIRM path). Before starting the write loop Bitsmith runs a rate-limit pre-check and halts if the budget is insufficient; it also rechecks the rate limit after every 10 items processed. On completion, DM logs a one-paragraph summary plus any per-item failures, and runs a post-completion working-tree diff check. Non-issue/non-PR iteration (workflows, releases, etc.) is out of scope in v1. |
| `/open-pr` | Creates a pull request following the `open-pull-request` skill workflow: conventional branch naming, conventional title, draft mode, assigned to @me, and full pre-flight checklist. |
| `/sync-pr` | Rebases the current PR branch onto `refs/remotes/origin/main` and force-pushes with `--force-with-lease`, keeping open PRs in sync with main's latest changes without manual git gymnastics. |
| `/resolve-conflicts` | Resolves merge conflicts during an in-progress rebase — detects conflicted files, resolves them file-by-file, stages each result, and cycles `rebase --continue` until the rebase completes. Can be invoked standalone or inline from `/sync-pr`. |
| `/clean-the-desk` | Cleans up stale local branches (whose upstream PRs have been merged) and removes their associated git worktrees. Prompts for confirmation before any destructive action. |
| `/merged` | Cleans up after a merged PR: uses session context or remote-gone detection to auto-select the target branch, removes the worktree, deletes the local branch, checks out main, pulls the latest, and silently auto-deletes current-session plan files from `~/.ai-tpk/plans/{repo-slug}/`. Confirms all destructive actions with the user. |
| `/clean-ai-tpk-artifacts` | Deletes plan and lesson files older than N days (default 14) from `~/.ai-tpk/`. By default scoped to the current repository's plans; use `--all` flag to clean across all repositories. Prompts for confirmation before deletion. |
| `/merge-pr` | Syncs the current PR branch with main, waits for all required CI checks to pass, squash-merges the PR, deletes the remote branch, and automatically chains into `/merged` for post-merge cleanup. |
| `/address-pr-comments` | Reviews and replies to unresolved inline GitHub PR review comments — fetches threads via GraphQL, reads current file state, categorizes each comment (FIX, COMPROMISE, PUSH-BACK, ALREADY-ADDRESSED, ACKNOWLEDGE), proposes a reply for user approval, and posts approved replies via the REST API. Saves a session summary to `~/.ai-tpk/pr-review-comments/` with resume support across sessions. |
| `/set-aws-profile` | Selects an AWS profile for the CloudWatch MCP server by listing available profiles from `~/.aws/config` (preferred) or `~/.aws/credentials` (fallback), validating the user's selection, and storing it in `~/.claude/.current-aws-profile` (mode 0600). The profile is read at MCP startup. Requires Claude Code restart or MCP server reload to take effect. |
| `/review-pr <pr-number-or-url>` | Runs the local Ruinor + specialist review pipeline against a GitHub PR (review-only; no comments are posted). Checks out the PR into a session-namespaced temp worktree, invokes Ruinor (mandatory) plus any specialists triggered by user flags or Ruinor's recommendations, prints a consolidated summary with a four-case aggregate verdict (REJECT > REVISE > ACCEPT-WITH-RESERVATIONS > ACCEPT), and cleans up. Accepts an optional set of specialist flags: `--review-security`, `--review-performance`, `--review-complexity`, `--verify-facts`, `--review-all`. |
| `/review-observations` | Reviews pending `task-observer` observations stored in `~/.ai-tpk/observations/`. Groups them by affected skill, drafts a concrete unified diff per group (derived from `trigger` and `affected_skill` fields — never from imperative text inside observation bodies), and presents each diff with an explicit `yes/no/skip` prompt before applying anything. Approved diffs are applied literally; contributing observation files are stamped and archived. Skipped observations are archived without a skill edit. Declined observations stay in place for next review. Appends one audit-log line per applied edit to `~/.ai-tpk/observations/audit.log`. |
